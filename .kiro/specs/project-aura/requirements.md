# Requirements Document

## Introduction

Project Aura is an AI-powered framework for continuity of care in hospice settings, built on a four-pillar architecture that ensures comprehensive, ethical, and compassionate care throughout the entire patient journey—from diagnosis through end-of-life care and into bereavement support.

**The Four Pillars:**

1. **Pillar 1: Intelligent Core (XAI Engine)** - Explainable AI for transparent care pathway recommendations using XGBoost and SHAP
2. **Pillar 2: Safety Layer** - Structured Support Hub and Bereavement Bridge providing safe, curated interactions without open-ended AI risks
3. **Pillar 3: Human Bridge (Care Dashboard)** - Integrated Streamlit web interface with proactive alerts and role-based access
4. **Pillar 4: Validation Engine (Scenario Simulator)** - Synthetic data generation and end-to-end journey simulation using SDV

The system addresses critical gaps in current hospice care technology: opaque AI decision-making, fragmented tools, lack of post-death support, and reactive rather than proactive monitoring. Project Aura emphasizes compassion, dignity, transparency, and privacy through explainable AI, unified dashboards, structured support systems, and exclusive use of synthetic data.

## Glossary

- **Project_Aura**: The complete AI-powered hospice care framework system built on four-pillar architecture
- **Pillar_1_Intelligent_Core**: The XAI Engine providing explainable AI recommendations
- **Pillar_2_Safety_Layer**: The structured Support Hub and Bereavement Bridge ensuring safe interactions
- **Pillar_3_Human_Bridge**: The Care Dashboard providing integrated interface and proactive alerts
- **Pillar_4_Validation_Engine**: The Scenario Simulator generating synthetic data and validating system integration
- **XAI_Engine**: Explainable AI Engine that generates transparent care recommendations using XGBoost and SHAP interpretability
- **Care_Dashboard**: The integrated Streamlit web-based interface for multi-user logging, trend visualization, and alert management
- **Support_Hub**: The structured, menu-driven interface for symptom logging and resource access (no open-ended AI)
- **Bereavement_Bridge**: Post-death support module providing digital memories, grief journaling, and curated resources
- **Alert_System**: Proactive notification system that monitors patient trends and triggers email notifications via SendGrid
- **Scenario_Simulator**: Validation engine that generates synthetic patient journeys using SDV and validates end-to-end system functionality
- **Synthetic_Data**: Artificially generated patient data created using SDV (Synthetic Data Vault) to ensure privacy and enable bias-free training
- **SHAP**: SHapley Additive exPlanations - XAI technique for visualizing feature importance in ML predictions
- **Care_Pathway**: Personalized treatment and symptom management plan recommended by the XAI_Engine (HHC, HIC, HSC, or crisis intervention)
- **Clinician**: Healthcare professional user (doctor, nurse, hospice worker) with full system access
- **Family_User**: Patient family member or caregiver with restricted, patient-specific system access
- **Patient_Journey**: Complete care continuum from diagnosis through monitoring, alerts, death, and bereavement
- **SDV**: Synthetic Data Vault - library for generating realistic synthetic datasets with diversity constraints

## Requirements

### PILLAR 1: INTELLIGENT CORE (XAI Engine)

### Requirement 1: Explainable AI Engine

**User Story:** As a clinician, I want transparent AI-powered care recommendations with visual explanations, so that I can understand the reasoning behind suggestions and make informed decisions with confidence.

#### Acceptance Criteria

1. WHEN THE XAI_Engine receives patient data (vitals, symptoms, pain levels), THE XAI_Engine SHALL generate a personalized care pathway recommendation using trained machine learning models
2. WHEN THE XAI_Engine produces a recommendation, THE XAI_Engine SHALL generate SHAP-based visual explanations showing feature importance for that specific prediction
3. WHEN a clinician views a recommendation, THE Care_Dashboard SHALL display both the care pathway and the corresponding SHAP visualization (feature importance plot)
4. WHEN THE XAI_Engine trains models, THE XAI_Engine SHALL use only synthetic patient data generated by the Scenario_Simulator
5. WHEN THE XAI_Engine makes predictions, THE XAI_Engine SHALL calculate deterioration risk scores alongside care pathway recommendations

### PILLAR 4: VALIDATION ENGINE (Scenario Simulator)

### Requirement 2: Synthetic Data Generation and Bias Mitigation

**User Story:** As a system architect, I want diverse, realistic synthetic patient data, so that the system can be validated without privacy risks and AI models are trained without demographic bias.

#### Acceptance Criteria

1. WHEN THE Scenario_Simulator generates synthetic data, THE Scenario_Simulator SHALL use SDV library to create realistic patient records including demographics, vitals, symptoms, and timelines
2. WHEN generating patient demographics, THE Scenario_Simulator SHALL ensure diversity across age, gender, ethnicity, and socioeconomic indicators to mitigate training bias
3. WHEN synthetic data is created, THE Scenario_Simulator SHALL store it in local SQLite database without any real patient information
4. WHEN THE XAI_Engine accesses training data, THE XAI_Engine SHALL verify that all data originates from synthetic sources only
5. WHEN synthetic patient journeys are generated, THE Scenario_Simulator SHALL include complete care continuum stages (diagnosis, monitoring, deterioration, death, bereavement)

### PILLAR 3: HUMAN BRIDGE (Care Dashboard & Alerts)

### Requirement 3: Integrated Care Dashboard

**User Story:** As a family member, I want a unified dashboard where I can log symptoms, view trends, and receive alerts, so that I can actively participate in care coordination without juggling multiple tools.

#### Acceptance Criteria

1. WHEN a user accesses THE Care_Dashboard, THE Care_Dashboard SHALL display a role-appropriate interface based on user type (clinician or family member)
2. WHEN a user logs symptoms or vitals, THE Care_Dashboard SHALL store the entry in the database with timestamp and user attribution
3. WHEN viewing patient trends, THE Care_Dashboard SHALL display interactive charts showing vitals and symptom progression over time
4. WHEN THE Care_Dashboard is accessed, THE Care_Dashboard SHALL provide navigation to all system modules (logging, trends, alerts, support, bereavement)
5. WHEN THE Care_Dashboard renders, THE Care_Dashboard SHALL use empathetic design with soft colors, large readable text, and accessible controls

### Requirement 4: Proactive Alert System

**User Story:** As a clinician, I want automatic alerts when patient conditions deteriorate, so that I can intervene proactively rather than reactively discovering problems.

#### Acceptance Criteria

1. WHEN patient vitals or symptoms show deteriorating trends, THE Alert_System SHALL detect the pattern through background monitoring
2. WHEN a deterioration pattern is detected, THE Alert_System SHALL trigger email notifications to designated clinicians and family members
3. WHEN configuring alerts, THE Care_Dashboard SHALL allow users to set thresholds and notification preferences
4. WHEN THE Alert_System sends notifications, THE Alert_System SHALL include patient identifier, alert reason, and trend summary
5. WHEN THE Alert_System evaluates trends, THE Alert_System SHALL check database entries at regular intervals without blocking user interface operations

### PILLAR 2: SAFETY LAYER (Support Hub & Bereavement Bridge)

### Requirement 5: Structured Support Hub

**User Story:** As a family member, I want a safe, structured chat assistant for logging symptoms and finding resources, so that I can get help without the risks of open-ended AI conversations.

#### Acceptance Criteria

1. WHEN a user accesses THE Support_Hub, THE Support_Hub SHALL present a menu-driven interface with predefined symptom categories and resource topics
2. WHEN a user selects a symptom category, THE Support_Hub SHALL guide them through structured questions to log detailed symptom information
3. WHEN a user requests resources, THE Support_Hub SHALL provide curated, vetted information from a predefined knowledge base
4. WHEN THE Support_Hub processes user input, THE Support_Hub SHALL avoid open-ended generative AI responses to prevent harmful or inappropriate content
5. WHEN symptom logging is completed, THE Support_Hub SHALL save the structured data to the database for trend analysis

### Requirement 6: Bereavement Bridge Module

**User Story:** As a grieving family member, I want continued support after my loved one's death, so that I can access grief resources, preserve memories, and process my loss with guidance.

#### Acceptance Criteria

1. WHEN a patient death event is recorded, THE Bereavement_Bridge SHALL activate and become accessible to family users
2. WHEN THE Bereavement_Bridge is active, THE Bereavement_Bridge SHALL provide guided grief journaling prompts and forms
3. WHEN a family member accesses THE Bereavement_Bridge, THE Bereavement_Bridge SHALL display curated grief support resources including articles, hotlines, and coping strategies
4. WHEN a family member submits memories or journal entries, THE Bereavement_Bridge SHALL store them securely in the database
5. WHEN THE Bereavement_Bridge presents resources, THE Bereavement_Bridge SHALL organize content by grief stages and support types

### PILLAR 4: VALIDATION ENGINE (Continued)

### Requirement 7: Clinical Scenario Simulator

**User Story:** As a developer, I want to simulate complete patient journeys end-to-end, so that I can validate system integration and demonstrate functionality without real patient data.

#### Acceptance Criteria

1. WHEN THE Scenario_Simulator runs, THE Scenario_Simulator SHALL generate multiple synthetic patient journeys with varied characteristics
2. WHEN simulating a patient journey, THE Scenario_Simulator SHALL progress through all care stages (admission, monitoring, alerts, deterioration, death, bereavement)
3. WHEN a simulated journey includes deterioration, THE Scenario_Simulator SHALL trigger THE Alert_System to validate notification functionality
4. WHEN a simulated journey reaches death, THE Scenario_Simulator SHALL activate THE Bereavement_Bridge to validate post-death support
5. WHEN simulation completes, THE Scenario_Simulator SHALL generate summary reports with logs, visualizations, and validation results

### CROSS-PILLAR REQUIREMENTS (System Integration)

### Requirement 8: Role-Based Access Control

**User Story:** As a system administrator, I want role-based access controls, so that clinicians and family members see appropriate information and functionality for their roles.

#### Acceptance Criteria

1. WHEN a user logs in, THE Care_Dashboard SHALL authenticate credentials and assign appropriate role (clinician or family member)
2. WHEN a clinician accesses THE Care_Dashboard, THE Care_Dashboard SHALL display full patient data, AI recommendations, and administrative controls
3. WHEN a family member accesses THE Care_Dashboard, THE Care_Dashboard SHALL display patient-specific information with restricted access to sensitive clinical details
4. WHEN THE Care_Dashboard renders pages, THE Care_Dashboard SHALL show or hide features based on the authenticated user's role
5. WHEN user sessions are managed, THE Care_Dashboard SHALL maintain session state securely throughout the user's interaction

### Requirement 9: Privacy and Data Security

**User Story:** As a compliance officer, I want all patient data to be synthetic and stored locally, so that the system demonstrates functionality without HIPAA violations or privacy risks.

#### Acceptance Criteria

1. WHEN THE Project_Aura system operates, THE Project_Aura SHALL use only synthetic data generated by SDV library
2. WHEN data is stored, THE Project_Aura SHALL use local SQLite database without cloud transmission of patient information
3. WHEN THE XAI_Engine trains or makes predictions, THE XAI_Engine SHALL verify no real patient data is processed
4. WHEN audit logs are created, THE Project_Aura SHALL record all data access and AI decisions for transparency
5. WHEN THE Care_Dashboard displays data, THE Care_Dashboard SHALL include clear indicators that all data is synthetic for prototype purposes

### Requirement 10: System Integration and Deployment

**User Story:** As a developer, I want a cohesive system with all components integrated, so that users experience seamless functionality across all features.

#### Acceptance Criteria

1. WHEN THE Project_Aura launches, THE Care_Dashboard SHALL initialize all subsystems (XAI_Engine, Support_Hub, Alert_System, Bereavement_Bridge)
2. WHEN data flows between components, THE Project_Aura SHALL maintain consistency across database, ML models, and user interfaces
3. WHEN THE Care_Dashboard is deployed, THE Care_Dashboard SHALL run as a Streamlit web application accessible via browser
4. WHEN dependencies are installed, THE Project_Aura SHALL use only free, open-source libraries (scikit-learn, XGBoost, SHAP, SDV, Streamlit, SQLAlchemy)
5. WHEN THE Project_Aura is packaged, THE Project_Aura SHALL include documentation, setup instructions, and demo scripts in a Git-ready repository structure
